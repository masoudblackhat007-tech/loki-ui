# Section 16 - Add safe request logging for loki-ui

## Date

2026-05-07

## Goal

Add safe structured request logging to `loki-ui`.

The goal is to log useful operational request metadata without leaking sensitive request data.

## Problem

Before this section, `loki-ui` logged startup and shutdown events, but it did not log individual HTTP requests.

This made it harder to inspect service behavior during local testing, server operation, and future observability integration.

However, request logging can easily become dangerous if it records headers, cookies, authorization values, query strings, payloads, or response bodies.

## Security requirements

The request logger must not log:

- query string
- Cookie header
- Authorization header
- raw headers
- request body
- response body
- Loki URL
- raw Loki errors
- session values
- tokens

The request logger may log:

- request ID generated by `loki-ui`
- method
- path without query string
- status code
- duration in milliseconds
- fixed log type

## File changed

    internal/httpserver/server.go

## Implementation

A middleware was added around the HTTP mux:

    requestLogger(mux)

The HTTP server now uses:

    Handler: requestLogger(mux)

Each request gets an internal response header:

    X-Request-Id

The log entry is JSON encoded.

Example log shape:

    {
      "type": "loki_ui_request",
      "request_id": "11e87932a8d9689a883be3f67363b9e9",
      "method": "GET",
      "path": "/healthz",
      "status": 200,
      "duration_ms": 0
    }

The logger intentionally uses:

    r.URL.Path

It does not use:

    r.URL.String()

This prevents query strings from being logged.

## Local runtime validation

The binary was run locally on a temporary loopback port:

    LISTEN_ADDR=127.0.0.1:18091 LOKI_URL=http://127.0.0.1:3100 ./bin/loki-ui

Test requests intentionally included sensitive-looking query/header values:

    curl -i 'http://127.0.0.1:18091/healthz?token=SHOULD_NOT_APPEAR'

    curl -i 'http://127.0.0.1:18091/readyz?token=SHOULD_NOT_APPEAR'

    curl -i \
      -H 'Authorization: Bearer SHOULD_NOT_APPEAR' \
      -H 'Cookie: session=SHOULD_NOT_APPEAR' \
      'http://127.0.0.1:18091/healthz?secret=SHOULD_NOT_APPEAR'

Responses included generated request IDs:

    X-Request-Id

Observed safe request logs:

    {"type":"loki_ui_request","request_id":"11e87932a8d9689a883be3f67363b9e9","method":"GET","path":"/healthz","status":200,"duration_ms":0}
    {"type":"loki_ui_request","request_id":"14b55b6b1afeb344820914a3b96fe6c8","method":"GET","path":"/readyz","status":503,"duration_ms":2001}
    {"type":"loki_ui_request","request_id":"424044bf1d38b81daafddcdfa624bfa9","method":"GET","path":"/healthz","status":200,"duration_ms":0}

The sensitive-looking test strings did not appear in the request logs:

    SHOULD_NOT_APPEAR
    Authorization
    Cookie
    token
    secret

## Server validation commands

Generate a request:

    curl -i 'http://127.0.0.1:18090/healthz?token=SHOULD_NOT_APPEAR'

Inspect recent logs:

    journalctl -u loki-ui -n 30 --no-pager

Expected result:

    A JSON request log is present.
    The path is logged as /healthz.
    The query string is not logged.
    Sensitive-looking test values are not logged.

Network and firewall checks:

    ss -ltnp | grep 18090 || true
    sudo ufw status numbered

Expected security result:

    loki-ui must still listen only on 127.0.0.1:18090
    UFW must not expose 18090/tcp publicly

## Security notes

This request logger is intentionally minimal.

It is not an audit log.

It is not an authentication log.

It does not identify users.

It does not record headers or payloads.

If future authentication is added, this logger must be reviewed again before logging any identity-related field.

## Result

`loki-ui` now emits safe structured request logs with method, path, status, duration, request ID, and fixed log type.

The logger avoids query strings, headers, cookies, authorization values, and request bodies.

## Resume relevance

This section demonstrates adding security-conscious structured request logging to a Go internal observability service while intentionally avoiding sensitive data leakage.
